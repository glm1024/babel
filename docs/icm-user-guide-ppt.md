## 03 / 部署与运行

**本页功能**

- 说明系统当前以本地服务模式运行，用户只需要在本机启动服务并打开浏览器即可使用。
- 告诉用户结果文件会落到哪里，方便后续查看和留档。

**要填字段**

- 运行前提：`Python 3.6.1+`
- 启动命令：`PYTHONPATH=src python3 -m zh_audit serve`
- 可选命令：`PYTHONPATH=src python3 -m zh_audit serve --no-browser`

**操作步骤**

- 在项目根目录执行启动命令。
- 启动后系统会默认打开浏览器首页；如果不希望自动打开，可加 `--no-browser`。
- 结果文件默认写入 `results/`。

**注意事项**

- 这是本地服务，不是远程部署平台；演示时不要讲成服务器端系统。
- `results/` 下常见文件包括 `findings.json`、`summary.json`、`report.html`、`app_state.json`。

**右侧截图建议**

- 左半张截终端启动成功输出，右半张截浏览器打开后的首页。

## 04 / 首页：配置扫描目录

**本页功能**

- 首页的核心作用是配置待扫描项目目录，并发起一次扫描任务。
- 支持一次录入多个扫描目录，适合批量盘点多个仓库。

**要填字段**

- `扫描目录`：填写待扫描项目目录绝对路径。

**操作步骤**

- 在输入框中填写一个待扫描项目的绝对路径。
- 如果需要扫描多个项目，点击 `新增目录` 增加更多路径输入框。
- 路径确认无误后，点击 `开始扫描`。

**注意事项**

- 这里只接受目录路径，不是文件路径。
- 建议统一使用绝对路径，避免目录不存在或定位错误。

**右侧截图建议**

- 截首页上半部分，重点带出 `扫描目录`、路径输入框、`新增目录`、`开始扫描`。

## 05 / 首页：状态与进度

**本页功能**

- 展示扫描任务当前状态，帮助用户确认系统是否正在扫描、扫描到哪个项目、哪个文件。
- 扫描完成后，用户可从这里直接进入结果页。

**要填字段**

- 无，本页主要查看状态信息。

**操作步骤**

- 点击 `开始扫描` 后观察进度条变化。
- 关注 `当前项目`、`当前文件`、`开始时间` 等信息，判断扫描是否正常推进。
- 扫描完成后点击 `查看扫描结果` 进入结果页。

**注意事项**

- 首页只负责发起扫描和查看进度，不会自动修改任何源码内容。
- 如果状态异常或扫描失败，先回看扫描目录是否正确、目录是否真实存在。

**右侧截图建议**

## 06 / 扫描结果：先看无需整改

**本页功能**

- 先复核 `无需整改` 项，确认系统把“不需要改”的内容分对了。
- 这是结果复核的第一步，目的是先排除误判。

**要填字段**

- `动作`：切换到 `无需整改`
- 可选筛选项：`项目`、`分类`、`语言`、`关键字`

**操作步骤**

- 在结果页先把 `动作` 筛选切到 `无需整改`。
- 重点抽查 `代码注释`、`Swagger注解`、`普通文档`、`SQL脚本`、`Shell脚本`、`国际化文件`、`日志打印`、`测试与样例` 等分类。
- 判断这些内容是否真的可以保持不动。

**注意事项**

- `无需整改` 不代表一定正确，仍然要人工复核。
- 如果某条内容其实应该整改，就要回到源码里处理，不能因为系统判成 `无需整改` 就直接放过。

**右侧截图建议**

- 截结果页筛选区和表格，确保能看到 `动作` 下拉框已经切到 `无需整改`。

## 07 / 扫描结果：再看需要整改

**本页功能**

- 讲清用户真正要处理的重点是 `需要整改`。
- 告诉用户这些条目本质上都是中文硬编码，需要回源码手动修改。

**要填字段**

- `动作`：保持在 `需要整改`
- 可选筛选项：`项目`、`语言`、`关键字`

**操作步骤**

- 先看每条记录的文件位置、代码片段和分类线索。
- 回到对应源码位置修改中文硬编码。
- 修改完成后，如需在系统内做状态跟踪，可点击 `标记已整改`。

**注意事项**

- 结果表里会把所有整改项统一汇总成 `中文硬编码` 展示，但它们背后可能来自用户可见文案、错误提示、配置项、协议值、逻辑字面量或未知待确认。
- `标记已整改` 只是在结果页记录状态，不会自动修改源码；如有误点，可用 `重新打开` 恢复。

**右侧截图建议**

- 截结果表格中的整改项，并带出 `标记已整改` 按钮。

## 08 / 扫描结果：辅助能力

**本页功能**

- 展示结果页除了明细表之外的辅助能力，帮助用户更快定位问题和导出数据。
- 这一页主要解决“怎么看全局、怎么筛选、怎么带走结果”。

**要填字段**

- `项目`
- `动作`
- `分类`
- `语言`
- `关键字`
- `每页条数`、`页码`

**操作步骤**

- 先看左侧 `扫描摘要`，快速掌握扫描批次、项目范围、分类分布和跳过文件数量。
- 再用筛选区缩小范围，用分页快速浏览。

**注意事项**

- 建议先筛选再导出，便于按当前关注范围整理结果。
- `查看跳过详情` 适合排查为什么某些文件没有进入扫描，常见原因包括策略排除、二进制内容、文件过大或解码失败。

## 09 / 免改规则：作用与机制

**本页功能**

- 解释为什么要配置免改规则，以及它在扫描流程中的位置。
- 告诉用户这不是“修改当前结果”的工具，而是“影响后续扫描归类”的规则工具。

**要填字段**

- 无，本页先讲工作原理。

**操作步骤**

- 先从结果页找出那些反复出现、确认不需要整改的固定内容。
- 把这些稳定模式整理成免改规则。
- 保存后重新扫描，让系统在后续扫描中自动归入对应免改分类。

**注意事项**

- 免改规则会在 `无需整改` 下新增自定义子分类。
- 规则匹配范围是命中文本和代码片段；保存后只对后续重新扫描生效，不会回写当前已有结果。

**右侧截图建议**

- 截 `免改规则` 页整体，能同时看到左侧分组区和右侧规则编辑区。

## 10 / 免改规则：配置方法

**本页功能**

- 逐项说明免改规则页到底要填什么、怎么填、什么时候适合用关键字或正则。
- 让用户可以独立完成一条规则的新增、保存和验证。

**要填字段**

- `分类名称`：给这组免改规则起一个清晰名称。
- `启用该分类`：控制该分组是否参与后续扫描。
- `规则类型`：可选 `关键字` 或 `正则`。
- `关键字/正则表达式`：填写实际匹配内容。

**操作步骤**

- 点击 `新增分组`，先创建一个分类。
- 填写 `分类名称`，按需勾选 `启用该分类`。
- 点击 `新增规则`，选择 `关键字` 或 `正则`，再填写匹配内容。
- 点击 `保存规则`，然后回到首页重新扫描验证效果。

**注意事项**

- `关键字` 适合命中固定文本，`正则` 适合命中带变化的文本模式。
- 一个分组至少要有一条规则；分组名称建议保持唯一、易理解。

**右侧截图建议**

- 截规则分组编辑区域和单条规则编辑区域，重点带出 `规则类型` 与输入框。

## 11 / 文档翻译（PO/RST）

**本页功能**

- 解决 `.po` 文档和 RST 相关内容的中英翻译与校对问题。
- 支持 AI 生成候选译文、人工审批、继续任务和中途停止。

**要填字段**

- `待翻译 PO 文件绝对路径`
- `跳过手动审批，自动接受后续全部 AI 翻译`

**操作步骤**

- 填写待翻译 `.po` 文件绝对路径。
- 按需勾选自动接受，再点击 `开始校译`。
- 对待审批条目执行 `接受`、`重新生成`、`忽略`，或手动修改候选 `msgstr` 后再接受。
- 如任务中断，可使用 `继续任务`；如需暂停，可点 `停止任务`。

**注意事项**

- `.po` 路径必须指向一个已存在的文件，不能填目录。
- `接受` 后会直接覆盖原 `.po` 文件中的相邻 `msgstr`。
- 这一页会保护 RST 结构并应用术语约束；如果无法启动翻译，先到 `模型配置` 页检查参数是否完整。

## 12 / 配置翻译

**本页功能**

- 解决中英 `.properties` 国际化文件的补齐、校对和统一问题。
- 适合处理中英配置文件不同步、英文值缺失或历史翻译不一致的场景。

**要填字段**

- `中文国际化文件绝对路径`
- `英文国际化文件绝对路径`
- `跳过手动审批，自动接受后续全部 AI 翻译`

**操作步骤**

- 分别填写中文和英文 `.properties` 文件绝对路径。
- 按需勾选自动接受，点击 `开始校译`。
- 在待审批条目中选择 `接受`、`重新生成`、`忽略`，或手动修改候选英文后再接受。
- 处理过程中可用 `继续任务` 恢复中断任务。

**注意事项**

- 中文和英文路径都必须指向已存在的 `.properties` 文件，不能填目录。
- `接受` 后会直接写回英文 `.properties` 文件；如果英文文件里缺少该 key，系统会追加写入。
- 系统会强制使用 `resources/国际化专业术语词典.xlsx` 中的标准术语；若模型配置不完整，任务无法启动。

**右侧截图建议**

- 截顶部路径输入区和待审批条目区，能看到中文、英文路径输入框及审批按钮。

## 13 / SQL翻译

**本页功能**

- 解决 SQL 脚本中中文数据的英文翻译问题，输出可执行的英文更新 SQL。
- 适合批量把表中中文文案补成英文文案，而不是直接改原始脚本。

**要填字段**

- `SQL脚本所在目录绝对路径`
- `目标表名`
- `定位字段名`
- `中文文案字段名`
- `英文文案字段名`
- `当前建表 SQL（可选，留空时系统先尝试从当前目录自动推导）`
- `跳过手动审批，自动接受后续全部 AI 翻译`

**操作步骤**

- 先填写 SQL 目录、表名和中英文字段信息。
- 如表结构复杂，可手动粘贴建表 SQL；否则可留空让系统自动推导。
- 点击 `开始校译`，在待审批条目中执行 `接受`、`重新生成`、`忽略`，或手动修改候选英文后再接受。
- 接受后查看输出文件路径，按需复制和执行生成的 SQL。

**注意事项**

- `SQL脚本所在目录绝对路径` 必须真实存在，且 `目标表名`、`中文文案字段名`、`英文文案字段名` 为必填项。
- `定位字段名` 用于生成 `WHERE` 条件，不要求一定是主键；留空时按 `id` 处理。
- `接受` 不会修改原始 SQL 文件，而是会在同目录生成新的 `*_sql_audit_时间戳.sql` 文件，并追加 `UPDATE` 语句。

## 14 / 模型配置

**本页功能**

- 这是三类翻译能力的前置条件，用于配置可访问的 OpenAI-compatible 模型服务。
- 用户只有在这一页保存成功后，才可以稳定使用文档翻译、配置翻译和 SQL翻译。

**要填字段**

- `Base URL`
- `API Key`
- `模型名称`
- `Max Tokens`

**操作步骤**

- 在 `模型配置` 页填写模型服务地址、鉴权信息、模型名和最大输出长度。
- 点击 `保存模型配置`，等待系统完成连通性测试。
- 保存成功后，再返回各翻译页面执行 `开始校译`。

**注意事项**

- `供应商` 固定为 `openai compatible`，不需要手工修改。
- `Base URL` 支持填写主机根、`/v1` 或完整 `/v1/chat/completions`，保存时会自动归一化为 `/v1`。
- 配置优先级为：硬编码默认值 < 项目根目录 `zh-audit.config.json` < 本地 `results/app_state.json`。

**右侧截图建议**

- 截模型配置完整表单，确保能看到 `Base URL`、`API Key`、`模型名称`、`Max Tokens` 和 `保存模型配置`。
