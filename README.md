# babel

`babel` 是一个面向国际化治理的本地 Web 工具，用来批量扫描一个或多个代码仓库里的中文硬编码，辅助用户完成结果复核、整改跟踪，以及多种翻译任务。

它当前以本地服务模式运行，核心入口是浏览器中的 7 个页面：

- `首页`
- `扫描结果`
- `免改规则`
- `文档翻译`
- `配置翻译`
- `SQL翻译`
- `模型配置`

## 当前能力

- 扫描源码、模板、配置、注释、文档、SQL 脚本等文本内容中的中文命中。
- 按上下文把命中归类为需要整改或保持不动，并生成可离线查看的 `report.html`。
- 在结果页中按项目、动作、分类、语言、关键字筛选，支持导出 Excel、查看跳过文件详情。
- 支持把整改项标记为 `完成整改`，并把状态持久化后自动应用到后续扫描结果。
- 支持自定义 `免改规则`，通过关键字或正则把特定命中归到“无需整改”下的自定义子分类。
- 支持对中英 `.properties` 文件做 AI 校对和回写。
- 支持对 `.po` 文档做 AI 校对和回写，并保护 RST 结构。
- 支持从 SQL 脚本中提取中文数据，生成英文 `UPDATE` SQL。
- 支持在 UI 中维护 OpenAI-compatible 模型配置，也支持通过项目配置文件下发默认值。
- 保留 `validate` 校验模式，适合研发或规则维护者对已有结果做质量校验。

## 运行要求

- Python `3.6.1+`
- 不支持 Python 2

## 快速开始

### 启动本地服务

如果你在仓库源码目录里直接运行，推荐使用模块方式启动：

Linux or Mac:

```bash
PYTHONPATH=src python3 -m zh_audit serve
```

Windows PowerShell:

```powershell
$env:PYTHONPATH = "src"
python3 -m zh_audit serve
```

如果你已经把项目安装成命令行工具，也可以直接运行：

```bash
zh-audit serve
```

默认行为：

- 绑定 `127.0.0.1:8765`
- 输出本地访问地址
- 自动打开默认浏览器
- 支持 `results/` 为空时冷启动

常用参数：

```bash
PYTHONPATH=src python3 -m zh_audit serve --no-browser
PYTHONPATH=src python3 -m zh_audit serve --host 0.0.0.0 --port 9000
PYTHONPATH=src python3 -m zh_audit serve --config /absolute/path/to/zh-audit.config.json
```

参数说明：

- `--no-browser`：启动后不自动打开浏览器
- `--host`：指定监听地址，默认 `127.0.0.1`
- `--port`：指定端口，默认 `8765`
- `--config`：指定项目模型配置文件路径

### 日常使用顺序

推荐按下面的顺序使用：

1. 启动本地服务
2. 在 `首页` 填写一个或多个扫描目录绝对路径
3. 点击 `开始扫描`
4. 在 `扫描结果` 页先复核 `无需整改`，再处理 `需要整改`
5. 如有稳定误判场景，在 `免改规则` 中补规则后重新扫描
6. 需要翻译时，先确认 `模型配置` 正确，再进入 `文档翻译`、`配置翻译` 或 `SQL翻译`

## 页面说明

### 首页

首页用于发起扫描和查看扫描进度。

- 支持录入多个扫描目录
- 扫描目录必须是真实存在的目录绝对路径
- 点击 `开始扫描` 后会实时展示进度条、当前项目、当前文件、开始时间
- 扫描完成后可直接点击 `查看扫描结果`

扫描与翻译任务不能并发执行；任一翻译任务运行时，不能开始新扫描。

### 扫描结果

扫描结果页用于复核和跟踪整改状态。

主要能力：

- 左侧显示 `扫描摘要`、分类分布、跳过文件统计
- 右侧明细表支持按 `项目`、`动作`、`分类`、`语言`、`关键字` 过滤
- 支持分页
- 支持 `导出全部结果到 Excel`
- 支持 `查看跳过详情`

动作说明：

- `需要整改`：需要回源码手动修改的命中
- `保持不动`：当前判断为无需整改的命中
- `完成整改`：在结果页手动标记后的整改状态

补充说明：

- 结果页会把整改项统一汇总为 `中文硬编码`
- 点击 `标记已整改` 只会更新结果状态，不会自动修改源码
- 已标记状态会写入 `results/remediation_state.json`，并在后续扫描中自动尝试重新应用

### 免改规则

免改规则用于在“无需整改”下新增自定义子分类。

配置方式：

- 先新增分组
- 为分组填写 `分类名称`
- 通过 `启用该分类` 控制是否生效
- 为分组添加一条或多条规则
- 规则类型支持：
  - `关键字`
  - `正则`

匹配机制：

- 规则会匹配命中的文本和代码片段
- 命中后会把条目归入该自定义分类
- 保存规则后不会回写当前已有结果
- 必须重新扫描，规则才会应用到新的扫描结果

### 配置翻译

`配置翻译` 页面用于对中英 `.properties` 文件做校对和翻译。

输入项：

- `中文国际化文件绝对路径`
- `英文国际化文件绝对路径`
- `跳过手动审批，自动接受后续全部 AI 翻译`

行为说明：

- 两个路径都必须指向已存在的文件，不能指向目录
- 每条待审批记录都支持：
  - `接受`
  - `重新生成`
  - `忽略`
  - 手动修改候选英文后再接受
- `接受` 后会直接写回英文 `.properties` 文件
- 如果英文文件缺少对应 key，会自动追加写入
- 如服务重启或任务被停止，可通过 `继续任务` 恢复

补充说明：

- 若中英文文件里存在重复 key，任务会直接报错
- 翻译过程会检查占位符一致性，并强制应用术语词典中的标准英文

### 文档翻译

`文档翻译` 页面当前用于处理 `.po` 文件。

输入项：

- `待翻译 PO 文件绝对路径`
- `跳过手动审批，自动接受后续全部 AI 翻译`

行为说明：

- 路径必须指向已存在的 `.po` 文件
- 每条待审批记录都支持：
  - `接受`
  - `重新生成`
  - `忽略`
  - 手动修改候选 `msgstr` 后再接受
- `接受` 后会直接覆盖原 `.po` 文件中的相邻 `msgstr`
- 如服务重启或任务被停止，可通过 `继续任务` 恢复

补充说明：

- 系统会对 RST 结构做保护，不会把受保护片段直接交给模型自由改写
- 当术语词典中的 `模块` 为 `前端` 时，这部分词条可作为前端术语参与文档翻译

### SQL翻译

`SQL翻译` 页面用于从 SQL 脚本中提取中文数据并生成英文 `UPDATE` SQL。

输入项：

- `SQL脚本所在目录绝对路径`
- `目标表名`
- `定位字段名`
- `中文文案字段名`
- `英文文案字段名`
- `当前建表 SQL（可选，留空时系统先尝试从当前目录自动推导）`
- `跳过手动审批，自动接受后续全部 AI 翻译`

行为说明：

- `SQL脚本所在目录绝对路径` 必须真实存在
- `目标表名`、`中文文案字段名`、`英文文案字段名` 为必填项
- `定位字段名` 可留空，留空时默认按 `id` 处理
- 每条待审批记录都支持：
  - `接受`
  - `重新生成`
  - `忽略`
  - 手动修改候选英文后再接受

输出说明：

- `接受` 不会改原始 SQL 文件
- 系统会在输入目录下新生成一个文件：
  - `<table_name>_sql_audit_YYYYMMDD_HHMMSS.sql`
- 文件中会逐条追加 `UPDATE ... WHERE ...;`
- 如服务重启或任务被停止，可通过 `继续任务` 恢复

### 模型配置

三类翻译能力都依赖 `模型配置` 页面。

输入项：

- `Base URL`
- `API Key`
- `模型名称`
- `Max Tokens`

行为说明：

- `供应商` 固定为 `openai compatible`
- `Base URL` 支持填写：
  - 主机根，如 `http://127.0.0.1:8000`
  - `/v1` 地址
  - 完整 `/v1/chat/completions`
- 保存时会自动归一化到 `.../v1`
- 保存模型配置时会先做一次连通性测试

优先级：

1. 硬编码默认值
2. 项目根目录 `zh-audit.config.json`
3. `results/app_state.json` 中由 UI 保存的本地覆盖值

## 术语词典

项目默认术语词典文件路径为：

- `resources/国际化专业术语词典.xlsx`

翻译页面启动时都会读取它；如果文件缺失、格式错误或存在冲突词条，翻译任务会启动失败。

推荐表头格式：

- `模块, 中文, 英文`

兼容的简化表头：

- `中文, 英文`

规则说明：

- 整句命中词典时，系统会直接使用词典英文
- 句子中部分命中时，会把词条作为锁定术语约束模型输出
- `模块 = 前端` 的词条会作为前端术语参与文档翻译的特定场景

## 项目配置文件

如果你希望把默认模型配置随项目一起分发，可以在项目根目录放一个 `zh-audit.config.json`：

```json
{
  "model_config": {
    "base_url": "http://127.0.0.1:8000/v1",
    "api_key": "",
    "model": "deepseek-v3",
    "max_tokens": 4096
  }
}
```

UI 中保存的配置只会写入本地 `results/app_state.json`，不会回写项目配置文件。

## 结果与状态文件

服务运行中会按需生成这些文件：

- `results/findings.json`
- `results/summary.json`
- `results/report.html`
- `results/app_state.json`
- `results/remediation_state.json`
- `results/translation_session.json`
- `results/po_translation_session.json`
- `results/sql_translation_session.json`

说明：

- `findings.json`：扫描明细
- `summary.json`：扫描摘要
- `report.html`：离线 HTML 报告
- `app_state.json`：UI 保存的本地状态，包含扫描目录、免改规则、翻译配置和模型覆盖值
- `remediation_state.json`：用户在结果页标记的整改状态
- `translation_session.json`：配置翻译任务快照
- `po_translation_session.json`：文档翻译任务快照
- `sql_translation_session.json`：SQL 翻译任务快照

额外说明：

- 首次启动时 `results/` 可以为空
- `findings.json / summary.json / report.html` 会在首次扫描完成后生成
- `app_state.json` 会在保存扫描目录、规则或翻译配置后生成
- 翻译任务快照会在对应翻译功能使用后生成
- SQL 翻译生成的审计 SQL 文件不写入 `results/`，而是写回用户指定的 SQL 目录

## validate 模式

`validate` 模式适合研发或规则维护者对已有扫描结果做质量校验，不是普通用户的日常入口。

示例：

```bash
PYTHONPATH=src python3 -m zh_audit validate \
  --repo /absolute/path/to/repo \
  --summary /absolute/path/to/results/summary.json \
  --findings /absolute/path/to/results/findings.json
```

可选参数：

- `--out`：指定校验输出目录，默认使用 `summary.json` 所在目录
- `--scan-policy`：指定扫描策略覆盖文件

## 任务并发限制

服务同一时间只允许一个活动任务：

- 扫描运行时，不能启动任意翻译任务
- 任一翻译任务运行时，不能开始新扫描
- 三种翻译任务彼此之间也不能并发运行
